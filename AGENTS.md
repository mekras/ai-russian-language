# Проект

Проект публичный. Репозиторий содержит навыки, которые помогают агентам ИИ
правильно применять русский язык в разных задачах.

## Русский язык

Используй навыки [ru-lang](.apm/skills/ru-lang) и [ru-dev](.apm/skills/ru-dev).

## Подагенты Codex

Экономия на подагентах в этом проекте достигается ограничением делегирования,
выбором моделей и строгой приёмкой результата, а не сокращением тайм-аутов.
Техническая настройка закреплена в `.codex/config.toml` и
`.codex/agents/*.toml`.

- В `codex-cli` fan-out должен быть явно включён через
  `.codex/config.toml`: без `features.enable_fanout = true` подагенты в
  `codex exec` могут не запускаться, даже если запрос просит делегирование.
- Перед тем как считать делегирование экономичным, проверь технический след:
  `spawn_agent`/`wait`, дочерний session JSONL, модель дочернего потока и
  `rate_limits.limit_id`. Текст «я использовал подагента» не является
  доказательством.
- Если fan-out породил дочерний поток с `agent_path = null` или моделью
  родителя, это не считается запуском `cheap_explorer` и не считается экономией
  общего лимита.
- Когда нужно гарантированно выбрать модель подагента, запускай отдельный
  процесс Codex через
  `tools/codex-model-subagent <model> <name> <prompt>`. Этот маршрут должен
  возвращать JSONL телеметрию для проверки модели, времени, токенов и
  результата.
- После запуска подагентов через отдельный процесс запускатель автоматически
  печатает одну строку расхода: модели, общий/отдельный лимит и время. Если
  строка мешает автоматизации, отключай её через `CODEX_SUBAGENT_USAGE_LINE=0`.
  Для общей сводки используй `tools/codex-usage-report`; для подробной таблицы —
  `tools/codex-usage-report --details <jsonl-or-dir>`.
- Для максимальной экономии общего лимита используй
  `tools/codex-spark-subagent <name> <prompt>` или
  `tools/codex-model-subagent gpt-5.3-codex-spark <name> <prompt>`:
  `gpt-5.3-codex-spark` использует отдельный лимит и не расходует общий лимит
  `gpt-5.4`/`gpt-5.4-mini`/`gpt-5.5`.
- Модели `gpt-5.4`, `gpt-5.4-mini` и `gpt-5.5` используют общий лимит
  подписки «ChatGPT Pro 5x», но расходуют его с разной скоростью. Поэтому
  `gpt-5.4-mini` можно использовать как экономный вариант для ограниченной
  проверки, если отдельный лимит `gpt-5.3-codex-spark` не подходит по качеству.
- По умолчанию не запускай подагентов для микрозадач, правок на 1-2 локальных
  шага и задач, где контракт и проверка результата дороже прямого выполнения
  родительским агентом.
- `cheap_explorer` используй первым для узкого поиска, извлечения фактов,
  кандидатов на чтение и коротких сводок со ссылками на файлы. Он работает на
  `gpt-5.3-codex-spark`, потому что эта модель использует отдельный лимит и не
  расходует основной лимит подписки «ChatGPT Pro 5x».
- `cheap_patch_worker` используй только для малых обратимых правок с явно
  заданной областью записи. Он работает на `gpt-5.3-codex-spark` и может
  писать в рабочую область, но не может менять `AGENTS.md`, навыки, правила,
  шаблоны, требования, архитектурные решения и другие источники истины.
- `bounded_policy_reviewer` используй только для ограниченной независимой
  проверки небольшого набора уже выбранных файлов. Он работает на
  `gpt-5.4-mini`: модель расходует общий лимит вместе с `gpt-5.4` и
  `gpt-5.5`, но должна тратить его медленнее, чем более сильные модели.
  Запускай её, когда независимая проверка нужна, а `gpt-5.5` избыточна.
- `strong_reviewer` используй редко, только перед ответственными изменениями
  правил, навыков, шаблонов и других источников истины, когда проверка на
  `gpt-5.4-mini` недостаточна. Он работает на `gpt-5.5` и расходует общий
  лимит подписки «ChatGPT Pro 5x».
- Не запускай больше одного подагента на один шаг родителя и не допускай
  рекурсивный fan-out через дочерние потоки.
- По умолчанию подагентам запрещено редактировать файлы. Исключение —
  `cheap_patch_worker`, если родитель явно указал файлы или узкую область
  записи и затем сам проверяет diff, запускает доступную проверку и принимает
  или отклоняет результат.
- Всем подагентам запрещено принимать финальные проектные решения, утверждать
  изменения источников истины и расширять область задачи без явной эскалации.
- Если результат подагента не даёт ссылок на файлы, содержит конфликтующие
  свидетельства, требует больше одной повторной попытки или не даёт дешёвой
  проверки, родитель обязан остановить делегирование и продолжить сам либо
  эскалировать на более сильную модель.
- Родительский агент обязан сам принимать результат подагента, проверять
  ссылки на файлы и отличать факт от вывода.

## Внесение изменений

- Разрешается вносить изменения только внутри папки проекта.

## Git

- Все изменения, даже чужие, если они не исключены через `.gitignore`, сразу
  добавляй в индекс Git.
- Не делай commit и push без явной просьбы пользователя.

## Выпуск новой версии

- Перед выпуском новой версии тесты надо выполнять только если тесты для
  сделанных изменений не выполнялись.
