# ru-dev

Применяй русский язык аккуратно в задачах разработки ПО: сохраняй техническую
точность, учитывай терминологию проекта и пиши так, чтобы текст помогал
разработчику действовать.

## Обязательная зависимость

- Используй `ru-lang` как базовый навык для русского языка в прикладных и
  технических текстах.
- `ru-dev` является слоем правил для разработки ПО поверх `ru-lang`.
- Не дублируй в `ru-dev` общие правила русского языка, уже вынесенные в
  `ru-lang`.
- Если текущая среда явно показывает список навыков, проверь, что `ru-lang`
  доступен как самостоятельный навык.
- Если `ru-lang` недоступен, продолжай работу по правилам `ru-dev`.
  Учитывай, что базовый языковой слой отсутствует, и упоминай это только
  когда задача требует строгой редакторской проверки, установки навыков или
  диагностики качества навыка.
- Если правила расходятся, применяй их в таком порядке: явное требование
  пользователя, терминология проекта, смысл кода, затем рекомендации `ru-lang`
  и `ru-dev`.

## Когда применять

- Используй `ru-dev`, когда нужно сформулировать русский текст внутри этого
  процесса:
  сообщение коммита, пользовательскую строку в коде, комментарий в коде,
  описание изменения, отчёт по результатам работы, сообщение об ошибке,
  техническое решение или документацию разработки.
- Если задача включает подготовку, проверку, предложение или выполнение
  коммита, сначала применяй `ru-dev` к сообщению коммита даже тогда, когда
  черновик, команда или обсуждение даны не по-русски.
- Если задача включает выпуск версии, тег выпуска, повышение версии или
  подготовку команды `git commit` в любой форме (`-m`, `-F`, heredoc,
  `--amend`), сначала сформулируй русское сообщение коммита через этот навык,
  а уже потом выполняй Git-команду.
- Не используй `ru-dev` для общего русского текста без контекста разработки:
  в таком случае достаточно `ru-lang` или `ru-text`.

## Обязательный порядок

- Не русифицируй идентификаторы, имена API, команды, параметры, названия
  пакетов, классов, методов и файлов.
- Если создаёшь или правишь код с пользовательскими строками, по умолчанию
  делай этот вывод русскоязычным: тексты интерфейса, сообщения об ошибках,
  подсказки, уведомления, help и другие строки, которые увидит пользователь.
  Отступай от этого правила только если пользователь, проект или существующий
  контракт явно требуют другой язык.
- Если явно не указано иное, пиши комментарии в коде по-русски.
- Для сообщений коммитов используй форму результата, а не инфинитив или
  повелительное наклонение: `Добавлена проверка входных данных`,
  `Исправлена обработка пустого ответа`, `Обновлена документация`. Не пиши
  тему инфинитивом или повелительным наклонением: `Добавить проверку`,
  `Добавь проверку`, `Исправить обработку`.
- Перед `git commit` проверь тему: она должна быть глаголом совершённого вида
  в форме результата. Если тема начинается с инфинитива или повелительного
  наклонения, перепиши её до выполнения команды.
- Пиши тело сообщения коммита, когда оно поясняет суть и причину изменения,
  и оформляй его по правилам `ru-lang`.
- Не добавляй в сообщение коммита строки соавторства `Co-Authored-By`.
- Для выпусков версий не используй шаблоны вроде `Release 0.1.0`,
  `Bump version`, `Version bump`. Пиши по-русски по фактическому действию:
  `Выпущена версия 0.1.0`, `Подготовлен выпуск 0.1.0`,
  `Обновлена версия до 0.1.0`.
- Для подробных правил перед созданием текста разработки читай
  [правила разработческого текста](./references/development-text.md).

## Ограничения

- Этот навык не заменяет правила проекта, архитектурные соглашения, порядок
  тестирования или правила безопасности.
- Используй правила проекта и среды для разработки, Git, проверки кода
  и внесения правок.
- Не добавляй комментарий только ради русскоязычности. Если код понятен без
  пояснения, комментарий не нужен.
